# Allo 能力包本地调试方案(合作方·无源码版)

> 面向:拿到 **Allo 安装版客户端**(不含源码)的合作开发团队(如「设计图助手」)。
> 结论先行:**不需要 Allo 源码就能完成后端工作流 + iframe 界面的全量本地调试**。安装版客户端自带完整的 Python 网关运行时,并原生支持「旁加载」你们正在开发的包;桌面模式下本地 API 无鉴权,curl / Postman / 你们自己的前端 dev server 都能直接打;iframe 型包(指南 §7)旁加载后**界面卡片直接出现在客户端里**。
> 配套文档:《Allo能力包开发指南(合作方版)》(协议与交付契约);本文只讲"怎么在本机跑起来、怎么排错"。

---

## 1. 原理(为什么不需要源码)

Allo 桌面端启动时,本地网关会扫描 `~/.allo/packs-active-backend/` 目录,把其中每个条目当作一个能力包后端挂载(读 `pack_backend.json` → 加 import 路径 → 挂路由)。这个目录里的条目**可以是软链接、Windows junction,也可以是普通目录**——正式 OTA 安装放的是链接,你们调试时手工放一个链接指向自己的开发目录,效果完全等价。

三个配套事实:

| 事实 | 含义 |
|---|---|
| 桌面模式网关无鉴权 | `http://127.0.0.1:8001` 上你们的路由可以直接 curl,不需要 token |
| 运行时 Python 固定 3.12 | mac 客户端自带解释器可直接用;win 为冻结 exe,调试依赖自备 Python 3.12(§4) |
| 挂载失败 fail-closed + 记日志 | 包坏了只会被跳过(客户端其他功能不受影响),原因写在网关日志里 |

### 1.1 预期管理(开工前必读的两句大实话)

1. **iframe 型包(v1.1 标准形态):旁加载连界面一起加载。** 只要 `pack.json` 声明了 `frontend.kind: "iframe"`(指南 §7),旁加载重启后,客户端「智能体 → 能力包」里会**直接出现你们的卡片**,点开就是你们的 UI——所见即正式安装后的体验,界面调试完全不依赖 Allo 方。两个前提:①`pack.json` 必须放在 `backend-ext` 的**上一级目录**(骨架仓库布局天然如此),网关按这个位置读包元数据,放错时包退化为"仅后端"(路由都在,卡片不出现);②客户端 ≥ 0.1.21(第一个带 iframe 宿主的版本)。只有旧的编译型前端(v1.0 形态,已弃用)才有"旁加载看不到界面"的问题。
2. **旁加载是调试通道,不是分发通道。** 员工正式装包走完全不同的路:OTA 下载 → `~/.allo/packs/<featureKey>/versions/<版本>/` 落盘 → 客户端自动建链接,且有功能组授权把关。旁加载只是把"建链接"这步手工做了,跳过下载与授权,**只影响开发者自己这台机器**——它永远不能用来给别人装包。

完整闭环一览:**骨架包起步 → 旁加载(后端路由 + iframe 卡片同时生效)→ curl 调 API、点卡片调界面 → `build_pack.py` 打 zip 交付 Allo 方发布 → 授权账号在客户端里正式安装——正式安装体验与旁加载看到的一致。**

## 2. 前提

- 一台装了 Allo 客户端的机器:**Mac(Apple Silicon,≥ 0.1.11)** 或 **Windows x64(≥ 0.1.19-beta.1,轻量版)**;**iframe 前端(指南 §7)两系统都需 ≥ 0.1.21**。两个系统的调试机制相同,只有链接命令和依赖安装方式有差异(§3/§4 分别给出)。
- 一个 Allo 测试员工账号(Allo 方提供)。**挂载和调路由不需要登录**;但若你们的工作流要调大模型(走 Allo 网关凭据),需先在客户端里完成员工登录。
- 骨架包 `pack-skeleton.zip`(Allo 方交付):解压后**不改任何东西**先按其 README 旁加载跑通一次(`/api/skeleton-demo/ping` 返回 200 + 客户端里出现「骨架示例」卡片),确认环境没问题,再开始改造成你们的包。
- 你们的开发目录,结构如下(featureKey 以 `design_assistant` 为例):

```
<你们的仓库>/
├── pack.json                      # 包元数据(iframe 前端声明;必须在 backend-ext 的上一级,§1.1)
└── backend-ext/
    ├── pack_backend.json          # 挂载声明(见开发指南 §3;静态路由排最后)
    ├── pydeps/                    # 调试用依赖目录(§4 生成;不进交付 zip)
    └── design_pack/               # 你们的 Python 包
        ├── __init__.py
        ├── routers/...            # 含 static.py(服务 frontend_dist,指南 §7.2)
        ├── services/...
        ├── frontend_dist/         # 构建好的 UI(iframe 形态)
        └── startup.py
```

## 3. 旁加载(一条软链)

macOS:
```bash
ln -s /绝对路径/你们的仓库/backend-ext ~/.allo/packs-active-backend/design_assistant
```

Windows(cmd;junction 类型,**无需管理员权限**):
```bat
mklink /J "%USERPROFILE%\.allo\packs-active-backend\design_assistant" "C:\绝对路径\你们的仓库\backend-ext"
```

注意:链接名**必须等于 featureKey**(小写字母/数字/下划线),链接目标是 `backend-ext` 目录本身。装载器对软链/junction/普通目录一视同仁——不想建链接的话直接把 `backend-ext` 拷贝成该目录也行,只是改代码后要重拷,不如链接省事。

## 4. 装依赖

你们独有的 Python 依赖(基座已有的库**不要**装,清单先和 Allo 方对表)装进 `backend-ext/pydeps/`:

macOS(客户端自带 Python 3.12,含 pip):
```bash
"/Applications/Allo Desktop.app/Contents/Resources/python/bin/python3" \
  -m pip install --target /绝对路径/backend-ext/pydeps -r 你们的requirements.txt
```

Windows(**客户端不带独立 Python**——轻量版后端是冻结的 `allo-backend.exe`,正式装包时它内置 wheels 安装;调试依赖用你们自己的 Python 3.12):
```bat
py -3.12 -m pip install --target C:\绝对路径\backend-ext\pydeps -r 你们的requirements.txt
```

- mac 用客户端自带解释器的好处:版本(3.12)和平台与真实运行时**严格一致**,不会出现"本机能跑、装进包不能跑"。
- Windows 上注意两点:①必须 Python **3.12**(带原生二进制的库 pip 会按解释器选 win_amd64 + cp312 轮子,和运行时一致);②别去 `resources\python` 找解释器,不存在。
- 两个系统都支持 `ALLO_PACKS_PYTHON` 环境变量显式指定装包解释器(一般用不到)。
- `pydeps/` 只服务调试。**正式交付 zip 里不放 pydeps**,只放 wheels(打包脚本负责),客户端安装时自己装。
- pydeps 的位置:标准放 `backend-ext/pydeps/`;装载器在旁加载这种"平铺"布局下也支持**向上搜索**——放在包根(`backend-ext` 的上一级,和 pack.json 同级)的 `pydeps/` 目录或指向别处的同名软链同样会被认到。正式 OTA 安装的 pydeps 位置由客户端管理,与此无关。

## 5. 挂载与验证

1. **完全退出** Allo(菜单栏退出,不是关窗口),重新打开——网关只在启动时扫包,改代码后同样靠重启生效。
2. 验证:

```bash
# 网关活着
curl http://127.0.0.1:8001/health
# 你们的路由挂上了(路径 = pack_backend.json 的 mountPrefix)
curl http://127.0.0.1:8001/api/design-workbench/<你们的某个GET端点>
# iframe UI 也挂上了(v1.1;entry 必须写到具体文件,指南 §7.3)
curl -I http://127.0.0.1:8001/api/design-workbench/ui/index.html
```

路由通了 = 后端调试环境就绪,之后就是普通的 FastAPI 开发循环:改代码 → 重启 Allo → curl。iframe 包此时客户端「智能体 → 能力包」里已能看到你们的卡片(§1.1);界面联调见 §7。

## 6. 排错(唯一通道:网关日志)

挂载失败**不会弹窗**(提示通道在路线图上),原因全部在网关日志里:

macOS:
```bash
tail -f ~/.allo/logs/desktop/gateway.log     # 实时看
grep -i design_assistant ~/.allo/logs/desktop/gateway.log   # 翻历史
```

Windows(同一目录结构):
```bat
type "%USERPROFILE%\.allo\logs\desktop\gateway.log"
findstr /i design_assistant "%USERPROFILE%\.allo\logs\desktop\gateway.log"
```

也可以从菜单栏 Allo 图标 →「打开日志目录」进同一个地方。常见跳包原因对照:

| 日志里的信号 | 原因 | 解法 |
|---|---|---|
| `pack_backend.json` 解析/schema 报错 | 声明文件写错 | 对照开发指南 §3 逐字段核 |
| import error / ModuleNotFoundError | 依赖没装进 pydeps,或 import 了基座不允许的内部模块 | §4 重装;import 面收敛到 `app.gateway.auth` + `deerflow.*` |
| module ... not inside declared package | 路由模块跑到声明的 package 目录外(防劫持规则) | 所有代码收进 `design_pack/` 之下 |
| route ... outside mountPrefix / 前缀冲突 | 路由 path 没落在 mountPrefix 下,或撞了别的包 | 全部路由统一前缀 |
| SystemExit | 模块里有脚本式 `sys.exit()` 守卫 | 去掉模块级副作用 |
| 日志里完全没有你的 featureKey | 链接没建对 / 没完全重启 | `ls -l ~/.allo/packs-active-backend/`;确认真退出了 |
| 路由都通、客户端却没卡片 | pack.json 不在 backend-ext 上一级 / 没写 `frontend.kind: "iframe"` / 客户端 < 0.1.21 | 对照 §1.1 与指南 §7.1 |
| 卡片有、点开"包 UI 未就绪" | 桥没照骨架抄(写死宿主源)或 UI 白屏 | 先浏览器直开 `<mountPrefix>/ui/index.html` 排除静态问题,再核指南 §7.4 |

**铁则:包坏了只会被整包跳过,永远不会带崩客户端**——所以放心大胆地试,日志是你们唯一要盯的东西。

## 7. 前端联调(v1.1:iframe 形态)

iframe 型包(指南 §7)的界面调试有三层,由浅入深:

1. **浏览器直开**:访问 `http://127.0.0.1:8001/<mountPrefix>/ui/index.html`——静态路由、相对资源路径、API 基址自定位一次验完。注意地址要写到 `index.html`(铁律 1,目录 URL 会被重定向"纠正");这个模式下没有宿主,桥消息静默降级,属正常。
2. **客户端里点卡片**:旁加载后卡片直接出现(§1.1),这是**唯一**能验证桥(`allo:ready`/`allo:init`/`allo:exit`/`allo:title`)与主题注入的方式。宿主 5 秒收不到 `allo:ready` 会显示"包 UI 未就绪"——此时先浏览器直开排除静态问题,再查桥 shim 是不是照骨架抄的(别写死宿主源,铁律 2)。
3. **UI 迭代循环**:`frontend_dist` 是静态文件,改完**刷新工作台即可**(index.html no-store),不用重启 Allo;只有后端 Python 改动才要重启。用 vite 的真实应用:`vite build` 输出进 `frontend_dist` 后刷新;开发期也可 `vite dev`(起在 `localhost:3000`,本地网关 CORS 已放行该源)配代理把 `/api` 转给 `http://127.0.0.1:8001` 快速迭代——但交付代码的 API 基址必须走自定位(铁律 3),别把 dev 写法带进产物。

## 8. Windows 与 macOS 的差异速查

Windows 能力包链路已验收(0.1.19-beta.1 起),两系统调试机制相同,差异就这四条:

| | macOS | Windows |
|---|---|---|
| 数据目录 | `~/.allo` | `%USERPROFILE%\.allo`(结构完全相同) |
| 旁加载链接 | `ln -s` | `mklink /J`(junction,无需管理员) |
| 调试依赖安装 | 客户端自带 `Resources/python/bin/python3` | **自备 Python 3.12**(客户端后端是冻结 exe,不带独立解释器) |
| 完全退出 | 菜单栏退出 | 退出后用任务管理器确认无 Allo 进程 |

你们的包代码本身跨平台,Mac 上调通的逻辑不需要为 Win 改动;唯一按平台分叉的是带原生二进制的依赖 wheel——交付打包时 `build_pack.py --fetch-wheels` 按目标平台自动抓,两个平台各出一个 zip,与你们的构建机系统无关。

## 9. 边界与纪律

- 旁加载**只影响你们自己这台机器**,和正式分发是两个世界:正式装包必须走 OTA + 功能组授权,员工侧看不到任何旁加载机制。
- 调完记得清场:mac `rm ~/.allo/packs-active-backend/design_assistant`;win `rmdir "%USERPROFILE%\.allo\packs-active-backend\design_assistant"`(junction 用 rmdir 只删链接不删你们的目录)→ 重启。
- 环境变量后门(一般用不到):`ALLO_PACKS_PYTHON` 可指定 pip 用的解释器;`ALLO_GLOBAL_DATA_DIR` 可整体挪 `~/.allo`。
- 模型调用走员工登录后的网关凭据,额度按测试账号所在功能组计;压测请提前打招呼。
